↻ End-to-End Workflow Steps

Phase 0

Message Ingestion & Validation (Steps 1–5)

1
User Sends Message in Teams
A staff member types a message in the JBS Assistant bot channel inside Microsoft Teams. The Azure Bot Service routes the message as a Bot Framework Activity object.
2
Bot Framework Delivers to Webhook
Azure Bot Service delivers an HTTPS POST /webhook/teams to the Webhook Service (port 8000) with an Authorization: Bearer <jwt> header.
Activity JSON includes: type, text, from, conversation, id, serviceUrl, timestamp
3
JWT Token Validation
Webhook validates the RS256 Bearer token using JWKS fetched from login.botframework.com.
  • Algorithm: RS256 (asymmetric, public key from JWKS)
  • Audience check: == TEAMS_APP_ID
  • Issuer check: https://api.botframework.com
  • Reject: missing token, wrong audience, invalid signature
4
Parse Teams Activity → NormalisedMessage
Only type="message" activities are processed. The event is normalized into a Pydantic NormalisedMessage object.
  • user_id — prefer aadObjectId, fallback to channel ID
  • text, timestamp (ISO-8601), reply_to (activity ID)
  • service_url, conversation_id
5
Forward to Orchestrator
Webhook POSTs the NormalisedMessage JSON to ORCHESTRATOR_URL/process via async httpx client with a 150-second timeout.
Phase 1

Code-Driven Data Collection — No LLM (Steps 6–10)

6
Deduplication Guard
Orchestrator tracks in-flight Teams activity IDs in an async-locked set. Exact re-deliveries (common after pod restarts) are silently dropped.
7
Load Session from StateManager
Session is loaded by user_id from the configured backend.
  • memory — in-process dict, lost on restart
  • sqlite — /tmp/jbs_sessions.db, restart-safe
  • external_redis — multi-replica deployments
  • Default TTL: 24 hours. New users get a fresh session (phase=1, step=1).
8
Stale Message Guard
Prevents queued/delayed messages from corrupting an active interview.
  • Drop reset commands (e.g. "new jbs") if message is >10 seconds old on active session
  • Drop any message >5 minutes old if session has collected fields
9
Phase 1 State Machine — 5 Steps, Pure Code
Each user message advances one step. No LLM is called. All replies are hard-coded templates.
1
Customer Name — greeting words trigger welcome message; any other input is saved as customer_name and advances to step 2.
2
Site Name — text saved as site_name, advance to step 3.
3
Site Category — parse "Aviation", "2", or "aviation" → normalize to one of 5 categories → bind collection_id from SITE_CATEGORY_COLLECTION_MAP.
4
Job Purpose — free-text, saved as job_purpose, advance to step 5.
5
Confirmation — display summary. Approval keywords (yes / confirm / ok) set phase1_confirmed=True and phase=2. "Change X" loops back to the relevant step.
10
Save Session & Send Teams Reply
Session is serialized and persisted. Bot Framework reply is sent to Teams using a cached OAuth2 client-credentials token (scope: https://api.botframework.com/.default).
Token cached in memory; refreshed 60 seconds before expiry.
Phase 2

RAG-Guided Content Collection — LLM + h2oGPTe (Steps 11–17)

11
suggest_duties — h2oGPTe RAG Call #1
Orchestrator calls H2OGPTeClient.chat() with the site-category collection bound in step 9-3.
Prompt: "List 3–5 typical duties for [job_purpose] at a [site_category] site. Output ONLY a numbered list."
Response parsed into a list of duty names, stored in p2_suggestions.duties.
12
confirm_duties — User Accepts or Overrides
User replies "confirm" (use suggestions) or types custom duties (one per line). Structured as duties = [{duty_name, tasks: []}]. State advances to suggest_tasks_0.
13
suggest_tasks_N — h2oGPTe RAG Call per Duty
One RAG call per duty. Loop index N tracks which duty is active.
Prompt: "List 3–5 tasks for duty '[duty_name]' in [job_purpose] at [site_category]. Format: task | frequency | responsible_role"
14
confirm_tasks_N — Parse & Store Task Rows
Each confirmed task is parsed into a structured object and stored under the duty.
Fields: sequence, task_description, trigger, frequency, responsible_role. Loop continues until all duties have tasks.
15
suggest_safety — h2oGPTe RAG Call for Safety
One RAG call for hazards and PPE requirements for the site type.
Prompt: "For a [site_category] site at [site_name], list typical hazards and required PPE. Format: Hazards: item1, item2 / PPE: item1, item2"
16
confirm_safety — Store Safety Data
User confirms or provides custom hazards/PPE. Stored as safety_compliance = {site_hazards: [...], ppe_requirements: [...]}. State advances to review.
17
review — Full JBS Summary & Approval Gate
Full JBS summary displayed. User can type APPROVE, "change duties", "change safety", or name a specific duty to re-enter.
On APPROVE → build_jbs_json() called and flow continues to Phase 3.
Phase 3

Document Generation & Delivery (Steps 18–24)

18
Build JBS JSON Payload
Orchestrator calls PhaseController.build_jbs_json() to assemble the complete structured payload from collected_fields.
Top-level fields: jbs_version, generated_at, metadata, duties[], safety_compliance
19
POST to Document Generator
Orchestrator sends the JBS JSON to DOCUMENT_GENERATOR_URL/generate (port 8002, internal) with a 60-second timeout.
20
Load Template & Replace Placeholders
Document Generator loads templates/jbs_corporate_template.docx and performs placeholder substitution.
Placeholders: {CUSTOMER_NAME} {SITE_NAME} {SITE_CATEGORY} {JOB_PURPOSE} {GENERATED_AT} {AUTHORIZED_BY}
21
Assemble Document Content
For each duty: add heading (level 2) + 5-column task table. Then add Safety & Compliance section. Append hidden audit trail.
  • Table columns: #, Task, Trigger, Frequency, Role
  • Safety paragraphs: Hazards, PPE
  • Audit trail: JSON in white 6pt font under "Appendix: Machine-Readable Data"
22
Upload to Azure Blob Storage
Generated file saved locally to /tmp/ then uploaded to Azure Blob Storage.
Filename: JBS_{site_name}_{uuid8}.docx
Container: jbs-documents
Prefix: jbs-documents/
23
Generate SAS URL & Return
A 15-minute read-only Shared Access Signature URL is generated and returned to the Orchestrator.
Response: {"download_url": "https://<account>.blob.core.windows.net/..."}
Expiry: DOC_URL_EXPIRY_SECONDS (default 900)
24
Final Teams Reply & Session Close
Orchestrator sends the download link to the user via Teams. Session status is set to "complete". User can type "New JBS" to restart.
Background

SharePoint → h2oGPTe Knowledge Base Sync

S1
Trigger
Runs automatically via Azure Container Apps Scheduled Job (daily 02:00 UTC) or on-demand via POST /admin/sync from the Wave dashboard.
S2
Per-Category Loop
For each of 5 site categories (Corporate, Aviation, Industrial, Maritime, Retail): get-or-create the h2oGPTe collection, list changed documents from the corresponding SharePoint library.
S3
Download & Ingest
Each changed document is downloaded via Microsoft Graph API, saved to a temp file, and uploaded to h2oGPTe for Document AI processing (OCR, chunking, embedding).
Chunks tagged with: site_category, doc_type, version. Available for RAG queries in Phase 2.
