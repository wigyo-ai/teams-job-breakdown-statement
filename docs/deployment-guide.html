Deployment Guide

Deploy the Job Breakdown Statement platform on Azure Container Apps. Three middleware services receive Microsoft Teams messages, run a 4-phase AI interview via H2O Enterprise h2oGPTe, and deliver a signed Word document via Azure Blob Storage.

☁️ Platform: Azure Container Apps
💬 Messaging: Microsoft Teams / Azure Bot Service
🤖 AI Engine: H2O Enterprise h2oGPTe (HAIC)
📦 Storage: Azure Blob Storage
🔄 CI/CD: GitHub Actions → ACR → ACA
🗺
Overview
Platform Architecture

Five components make up the platform. The three middleware services run on ACA and communicate via internal DNS. h2oGPTe and the admin dashboard remain on HAIC.

Service / ImagePortIngressEntrypointHosting
jbs-webhook 8000 External src.webhook.main:app Azure Container Apps
jbs-orchestrator 8001 Internal src.agent.server:app Azure Container Apps
jbs-docgen 8002 Internal src.document.server:app Azure Container Apps
certisjbs-sp-sync — Job python -m src.rag.sharepoint_sync ACA Scheduled Job (02:00 UTC)
jbs-dashboard 10101 HAIC wave run dashboard/app.py HAIC App Store — not ACA

Message flow: Teams → Azure Bot Service → Webhook (/webhook/teams, validates RS256 JWT) → Orchestrator (/process, 4-phase state machine, calls h2oGPTe) → on Phase 4 approval → Document Generator (/generate, renders .docx, uploads to Blob Storage) → Bot Framework REST API → reply in Teams.

4-phase interview: Phase 1 collects customer/site/category/purpose → Phase 2 collects duties → Phase 3 collects hazards/PPE/escalation → Phase 4 presents a summary for approval. Approval keywords: approve, confirm, yes, proceed, looks good.

Internal service URLs (within the ACA Environment): http://certisjbs-orchestrator and http://certisjbs-docgen — no port required within the same environment.

Set Up h2oGPTe Collections

The orchestrator selects a h2oGPTe collection per site category to ground all LLM responses in relevant SOPs. Five collections are required — one per category.

2a

Log in to h2oGPTe

Open your HAIC h2oGPTe instance URL in a browser and sign in with your HAIC credentials.

2b

Create one collection per site category

Navigate to Collections → New Collection and create:

collection_industrialSOPs and historical JBS for Industrial sites

⚠️

Record each Collection ID shown after creation. You will paste these into src/agent/phase_controller.py in Step 13a. The placeholders currently in the file (col_REPLACE_CORPORATE etc.) must be replaced before the orchestrator will route queries to the correct collection.

2c

Configure SharePoint as a document source

In h2oGPTe: Collections → [Select Collection] → Import → SharePoint Online. Enter Tenant ID, Client ID, Client Secret, Site URL, and Document Library. Enable Auto-sync → Daily. Repeat for all five collections.

SharePoint Azure AD App Registration

This app registration is used by src/integrations/graph_api_client.py to authenticate to the Microsoft Graph API and read SharePoint documents. It uses the OAuth 2.0 client credentials flow — no user login required.

✅

Step 5
Configure Microsoft Teams Bot

The Webhook service validates every incoming request against RS256 JWT tokens signed by Azure Bot Service (https://login.botframework.com/v1/.well-known/keys). The Orchestrator sends replies via the Bot Framework REST API using OAuth2 client credentials flow.

5a

Create an Azure Bot resource

  1. Azure Portal → search Azure Bot → Create
  2. Bot handle: certis-jbs-bot · Pricing tier: Standard
  3. Microsoft App ID: select Create new Microsoft App ID
  4. Review + create → Create
  5. Once deployed → navigate to the resource → Configuration
  6. Copy Microsoft App ID → this is TEAMS_APP_ID
  7. Click Manage → Certificates & secrets → New client secret → copy the value → this is TEAMS_APP_PASSWORD
5b

Enable the Microsoft Teams channel

Azure Bot → Channels → Microsoft Teams → Apply → accept Terms of Service.

5c

Set the messaging endpoint (after Step 11b)

Return here after Step 11b — you need the webhook Container App FQDN first.

url pattern
https://<certisjbs-webhook-fqdn>/webhook/teams

Paste this into Azure Portal → Azure Bot → Configuration → Messaging endpoint.

5d

Add the bot to your Teams tenant

Production: Create a Teams app manifest via Teams Developer Portal, set Bot ID to TEAMS_APP_ID, publish to your tenant app catalogue.

Testing: Teams → Apps → Manage your apps → Upload a custom app → upload the .zip package.

Step 6
Build Docker Images

Build all images locally. No Azure credentials required. The orchestrator image bundles config/prompts/; the docgen image bundles templates/.

bash
# Webhook — FastAPI on port 8000
docker build -f Dockerfile.webhook -t jbs-webhook:${IMAGE_TAG} .

# Orchestrator — FastAPI on port 8001 (includes config/prompts/)

# Document generator — FastAPI on port 8002 (includes templates/)
docker build -f Dockerfile.document -t jbs-docgen:${IMAGE_TAG} .

# H2O Wave dashboard — deployed via HAIC App Store, NOT pushed to ACR
docker build -f Dockerfile.dashboard -t jbs-dashboard:${IMAGE_TAG} .

Verify all four images built:

bash
docker images | grep jbs
💡

Test locally before pushing: docker compose up --build starts all three middleware services on ports 8000/8001/8002. Requires a populated .env. The compose file sets STATE_BACKEND=memory and wires inter-service URLs automatically.

Step 8
Create Azure Blob Storage

The Document Generator uploads .docx files here and returns a 15-minute SAS URL. Container name is certis-jbs-documents; expiry is controlled by DOC_URL_EXPIRY_SECONDS=900.

bash
az storage account create \
  --name $STORAGE_ACCOUNT \
  --resource-group $RG \
  --location $LOCATION \
  --sku Standard_LRS \
  --kind StorageV2 \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false

az storage container create \
  --name certis-jbs-documents \
  --account-name $STORAGE_ACCOUNT \
  --auth-mode login

# Capture the key — passed to Key Vault in Step 9
export STORAGE_KEY=$(az storage account keys list \
  --account-name $STORAGE_ACCOUNT \
  --resource-group $RG \
  --query "[0].value" -o tsv)

Update .env: set AZURE_STORAGE_ACCOUNT and AZURE_STORAGE_KEY to the values above.

Step 9
Create Azure Key Vault & Store Secrets

All four secrets used by the Container Apps are stored here. The Bicep template references them; for manual deployment they are passed as --secrets parameters directly to each Container App.

bash
az keyvault create \
  --name $KV_NAME \
  --resource-group $RG \
  --location $LOCATION \
  --sku standard \
  --enable-rbac-authorization true

az keyvault secret set --vault-name $KV_NAME --name teams-app-password  --value "<TEAMS_APP_PASSWORD>"
az keyvault secret set --vault-name $KV_NAME --name h2ogpte-api-key     --value "<H2OGPTE_API_KEY>"
az keyvault secret set --vault-name $KV_NAME --name azure-client-secret --value "<AZURE_CLIENT_SECRET>"
az keyvault secret set --vault-name $KV_NAME --name storage-key         --value "$STORAGE_KEY"

Step 12
Deploy SharePoint Sync as ACA Scheduled Job

The sync job reuses the orchestrator image and runs src/rag/sharepoint_sync.py daily at 02:00 UTC, ingesting changed SharePoint documents into h2oGPTe collections. Maximum runtime: 1800 seconds.

bash
az containerapp job create \
  --name certisjbs-sp-sync \
  --resource-group $RG \
  --environment $ACA_ENV \
  --trigger-type Schedule \
  --cron-expression "0 2 * * *" \
  --replica-timeout 1800 \
  --image ${ACR_LOGIN_SERVER}/jbs-orchestrator:${IMAGE_TAG} \
  --registry-server $ACR_LOGIN_SERVER \
  --registry-username $ACR_USERNAME \
  --registry-password $ACR_PASSWORD \
  --cpu 0.5 --memory 1.0Gi \
  --command "python" "-m" "src.rag.sharepoint_sync" \
  --env-vars \
    H2OGPTE_ADDRESS="https://your-h2ogpte-instance.h2o.ai" \
    AZURE_TENANT_ID="<AZURE_TENANT_ID>" \
    AZURE_CLIENT_ID="<AZURE_CLIENT_ID>" \
    SP_SITE_URL="https://certissecurity.sharepoint.com/sites/operations" \
    SP_LIBRARY_CORPORATE="<library-id>" \
    SP_LIBRARY_AVIATION="<library-id>" \
    SP_LIBRARY_INDUSTRIAL="<library-id>" \

Step 16
Production Hardening Checklist
  • All secrets stored in Azure Key Vault — no plaintext credentials in env-var definitions or terminal history
  • Webhook has external ingress; orchestrator and docgen have internal-only ingress
  • Teams Bot RS256 JWT validation confirmed — send a request without a Bearer token, expect HTTP 401
  • Teams app manifest published to tenant app catalogue (not side-loaded)
  • Azure Blob Storage public access disabled (--allow-blob-public-access false)
  • SAS URL expiry set to 15 minutes (DOC_URL_EXPIRY_SECONDS=900)
  • SharePoint App Registration limited to minimum permissions: Files.Read.All, Sites.Read.All only
  • Azure Key Vault RBAC: Key Vault Secrets User role assigned to ACA managed identities only
  • ACR admin credentials rotated post-deployment — switch to managed identity image pull for production
  • Container App min/max replica counts reviewed per service load profile
  • h2oGPTe collection IDs replaced in src/agent/phase_controller.py and orchestrator redeployed
  • Corporate Word template baked into docgen image (templates/jbs_corporate_template.docx)
  • SharePoint sync job initial manual run completed and verified
  • GitHub Actions CI/CD five secrets configured and a test push to main completed successfully
  • H2O MLOps monitoring configured for h2oGPTe inference latency and error rates

Troubleshooting

Teams bot not responding to messages

  1. Confirm webhook is running: curl https://${WEBHOOK_FQDN}/health
  2. Check messaging endpoint shows a green status in Azure Bot → Configuration
  3. Confirm TEAMS_APP_ID matches the Microsoft App ID in the Azure Bot resource (not the resource name)
  4. Check webhook logs:
bash
az containerapp logs show --name certisjbs-webhook --resource-group $RG --follow

5. Confirm orchestrator is reachable from webhook:

bash
az containerapp exec --name certisjbs-webhook --resource-group $RG \
  --command "curl http://certisjbs-orchestrator/health"

Webhook returns HTTP 401

  1. Confirm TEAMS_APP_ID is the Azure Bot's Microsoft App ID — copy it from Azure Bot → Configuration
  2. Verify the Bot Framework JWKS endpoint is reachable from the webhook container:
bash
