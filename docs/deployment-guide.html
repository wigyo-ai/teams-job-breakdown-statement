5-Phase Conversation Flow

1

Context & Initiation

Capture: Customer Name, Site Name, Site Category, Job Purpose → load site-specific RAG collection

2

Duty Discovery & Task Sequencing

Suggest tasks from SharePoint SOPs. Capture sequence, trigger, frequency, responsible role, outcome.

3

Safety & Compliance

Capture hazards, PPE, qualifications, accreditations, escalation procedures, comms channels.

4

Mozart Integration

Link reference documents (SOPs, Emergency Plans) from Mozart via Site ID and Document IDs.

5

Review & Document Generation

Present summary for approval → emit JSON → generate Word .docx → send download link.

Data Flow

User (Microsoft Teams) │ HTTPS POST — Bot Framework Activity │ JWT Bearer token (RS256, signed by Azure Bot Service) ▼ Webhook Service (FastAPI · HAIC) ← JWT validated │ ▼ Conversation Orchestrator (Python · HAIC) │ load session from SQLite │ build prompt (phase + RAG context + h2oGPTe history) ▼ H2O Enterprise h2oGPTe ◄──── SharePoint Collections (per site category) │ (Corporate / Aviation / Industrial / Maritime / Retail) │ LLM response ▼ Phase 4 only ──► Mozart API ──► reference document metadata │ Phase 5 approved ──► JSON ──► Document Generator ──► .docx ──► S3 │ ▼ Reply + download link ──► User (Bot Framework REST API · OAuth2 token)

Clone and Configure

Set up the project and configure environment variables
bash
# Clone the repository
git clone https://github.com/certis/jbs-platform.git
cd jbs-platform

# Copy the environment template
cp config/env.template .env

Edit .env and fill in all values. Key variables to configure:

bash — .env
# H2O Platform
H2OGPTE_ADDRESS=https://your-h2ogpte-instance.h2o.ai
H2OGPTE_API_KEY=<from H2O Secret Manager>

# Microsoft Teams (Azure Bot Service)
TEAMS_APP_ID=<Azure Bot App Registration client ID>
TEAMS_APP_PASSWORD=<Azure Bot App Registration client secret>

# Mozart
MOZART_API_BASE_URL=https://mozart.certis.internal/api/v1
MOZART_API_KEY=<from Mozart admin>

# SharePoint / Azure AD
AZURE_TENANT_ID=<your Azure tenant ID>
AZURE_CLIENT_ID=<app registration client ID>
AZURE_CLIENT_SECRET=<app registration client secret>
SP_SITE_URL=https://certissecurity.sharepoint.com/sites/operations

# SharePoint library IDs (get from Graph API Explorer)
SP_LIBRARY_CORPORATE=<library ID>
SP_LIBRARY_AVIATION=<library ID>
SP_LIBRARY_INDUSTRIAL=<library ID>
SP_LIBRARY_MARITIME=<library ID>
SP_LIBRARY_RETAIL=<library ID>

# Document storage
S3_BUCKET=certis-jbs-documents
S3_ENDPOINT_URL=https://storage.your-haic-cluster.h2o.ai
AWS_ACCESS_KEY_ID=<key>
AWS_SECRET_ACCESS_KEY=<secret>

# Internal
TENANT_ID=certis
ORCHESTRATOR_URL=http://jbs-orchestrator-service:8001
⚠️ Never commit .env to version control. It is included in .gitignore. Use H2O Secret Manager for all production secrets (Step 7).

Set Up h2oGPTe Collections

Create per-site-category knowledge base collections to power RAG

2a — Log in to h2oGPTe

Open your h2oGPTe instance URL in a browser and log in with your HAIC credentials.

2b — Create Collections via h2oGPTe UI

Navigate to Collections → New Collection and create one collection per site category:

Collection NameDescription
collection_corporateSOPs and historical JBS for Corporate sites
collection_aviationSOPs and historical JBS for Aviation sites
collection_industrialSOPs and historical JBS for Industrial sites
collection_maritimeSOPs and historical JBS for Maritime sites
collection_retailSOPs and historical JBS for Retail sites
ℹ️ Copy each Collection ID from the UI — you will need these to update SITE_CATEGORY_COLLECTION_MAP in src/agent/phase_controller.py (Step 12a).

2c — Configure SharePoint Integration in h2oGPTe

In h2oGPTe, navigate to Collections → [Select Collection] → Import → SharePoint Online and enter:

  • Tenant ID — your Azure AD tenant ID
  • Client ID — Azure app registration client ID
  • Client Secret — Azure app registration client secret
  • Site URL — SharePoint site URL
  • Document Library — select the appropriate library per category

Enable Auto-sync and set frequency to Daily.

2d — Manual Sync (alternative)

bash
# From project root, with .env loaded
python -m src.rag.sharepoint_sync
3

Configure Azure AD App Registration

Grants the platform read access to SharePoint via Microsoft Graph API
ℹ️ Skip this step if your Azure AD app registration is already configured with Graph API permissions.
  1. Log in to Azure Portal → Azure Active Directory → App registrations → New registration
  2. Set Name: CertisJBSPlatform · Supported account types: Single tenant → click Register
  3. Under API permissions, add Application permissions:
    Microsoft Graph → Files.Read.All
    Microsoft Graph → Sites.Read.All
  4. Click Grant admin consent
  5. Under Certificates & secrets → New client secret — copy the secret value into .env as AZURE_CLIENT_SECRET
4

Configure Microsoft Teams Bot (Azure Bot Service)

Create the Azure Bot resource and connect it to Microsoft Teams

4a — Create an Azure Bot resource

  1. Log in to Azure Portal → search for Azure Bot → Create
  2. Fill in: Bot handle = certis-jbs-bot, Pricing tier = Standard
  3. Under Microsoft App ID select Create new Microsoft App ID
  4. Click Review + create → Create
  5. Once deployed, navigate to the resource → Configuration
  6. Copy the Microsoft App ID → TEAMS_APP_ID
  7. Click Manage → Certificates & secrets → New client secret → copy value → TEAMS_APP_PASSWORD

4b — Set the messaging endpoint

In the Azure Bot resource → Configuration, set:

Messaging endpoint
https://<your-jbs-webhook-host>/webhook/teams
⚠️ The webhook service must be deployed (Step 7) and publicly accessible over HTTPS before Azure Bot Service will verify the endpoint. You can enter the URL now and confirm connectivity after deployment.

4c — Enable the Teams channel

  1. In the Azure Bot resource → Channels
  2. Click Microsoft Teams → Apply and accept the Terms of Service

4d — Add the bot to your Teams tenant

  1. Create a Teams app manifest using Teams Developer Portal
  2. Set the Bot ID to your TEAMS_APP_ID
  3. Upload to your tenant's app catalogue or side-load for testing:
    Teams → Apps → Manage your apps → Upload a custom app

7

Store Secrets in H2O Secret Manager

All sensitive credentials must be stored in HAIC — never in plaintext config
🔐 Never store secrets in values.yaml or any file committed to version control. HAIC Secret Manager is the only approved store for production credentials.
bash — h2octl
h2octl secret create --name h2ogpte-api-key        --value "<your key>"
h2octl secret create --name teams-app-password    --value "<your Teams bot client secret>"
h2octl secret create --name whatsapp-access-token  --value "<your token>"
h2octl secret create --name telegram-bot-token     --value "<your token>"
h2octl secret create --name mozart-api-key         --value "<your key>"
h2octl secret create --name azure-client-secret    --value "<your secret>"
h2octl secret create --name redis-password         --value "<your password>"
h2octl secret create --name s3-secret-access-key   --value "<your key>"

The Helm chart references these secrets by name via secretKeyRef — no plaintext values appear in the deployed manifests.

8

Deploy via Helm

Install all platform services into the HAIC Kubernetes cluster
ℹ️ This chart deploys 3 services only: Webhook, Orchestrator, Document Generator (+ CronJob). The Admin Dashboard is deployed via HAIC App Store (Step 9). Redis is eliminated — h2oGPTe owns conversation history natively.

8a — Edit values.yaml

Update all <PLACEHOLDER> values in deploy/helm/values.yaml:

yaml — values.yaml (key sections)
global:
  registry: registry.your-haic-cluster.h2o.ai/certis
  imageTag: "1.0.0"
  tenantId: "certis"
  orchestratorUrl: "http://jbs-orchestrator-service:8001"

webhook:
  replicaCount: 2
  ingress:
    host: jbs-webhook.your-haic-cluster.h2o.ai
    tlsSecret: jbs-webhook-tls
  env:
    plain:
      TEAMS_APP_ID: "<your bot app ID>"
    secret:
      TEAMS_APP_PASSWORD:
        name: certis-jbs-secrets
        key: teams-app-password

orchestrator:
  replicaCount: 2
  env:
    plain:
      H2OGPTE_ADDRESS: "https://your-h2ogpte-instance.h2o.ai"
      AZURE_TENANT_ID: "<tenant ID>"
      MOZART_API_BASE_URL: "https://mozart.certis.internal/api/v1"
      TEAMS_APP_ID: "<your bot app ID>"
      # No Redis config needed — h2oGPTe handles conversation history
      STATE_BACKEND: "sqlite"     # phase state only (tiny payload)
      SQLITE_PATH: "/data/jbs_sessions.db"
    secret:
      TEAMS_APP_PASSWORD:
        name: certis-jbs-secrets
        key: teams-app-password

syncCronJob:
  schedule: "0 2 * * *"   # Daily at 02:00 UTC

8b — Install with Helm

bash
# Create the Kubernetes namespace
kubectl create namespace certis-jbs

# Deploy all services
helm install certis-jbs deploy/helm/ \
  --namespace certis-jbs \
  --values deploy/helm/values.yaml \
  --wait
✅ The --wait flag holds until all pods reach Running state. Typically takes 2–4 minutes on first deploy.

Production Hardening Checklist

Complete all items before going live — click to mark as done
  • All secrets stored in H2O Secret Manager — no plaintext in values.yaml
  • Session state backend confirmed: sqlite for single-replica, external_redis (managed service) for multi-replica — no Redis pod deployed via Helm
  • Ingress TLS certificates issued and valid for both webhook and dashboard hosts
  • Teams Bot RS256 JWT Bearer token validation tested — invalid token returns HTTP 401
  • Teams Bot app manifest published to tenant app catalogue (not just side-loaded)
  • HAIC RBAC configured — only authorised roles access admin dashboard
  • S3 bucket policy restricts access to document generator service account only
  • SharePoint App Registration uses minimum permissions: Files.Read.All, Sites.Read.All
  • Mozart API key rotated from default
  • MLOps monitoring alerts configured and test alerts fired successfully
  • Pod resource limits set in Helm values (CPU / memory)
  • Horizontal Pod Autoscaler (HPA) configured for webhook and orchestrator
  • Document download URLs use short-lived presigned S3 URLs (15-minute expiry)
  • Penetration test of public webhook endpoint completed

Troubleshooting

Common issues and how to resolve them
💬 Bot not responding to messages
  1. Check webhook service logs:
    bash
    kubectl logs -l app=jbs-webhook -n certis-jbs --tail=50
  2. Verify the Teams messaging endpoint shows green status in Azure Portal → Azure Bot → Configuration
  3. Confirm TEAMS_APP_ID in the webhook env matches the Azure Bot App Registration
  4. Confirm orchestrator is reachable from the webhook pod:
    bash
    kubectl exec -it deploy/certis-jbs-webhook -n certis-jbs -- \
  curl http://jbs-orchestrator-service:8001/health
🧠 RAG returning no context / responses hallucinating
  1. Verify collections are populated:
    python
    python -c "from src.rag.h2ogpte_client import H2OGPTeClient; c=H2OGPTeClient(); print(c.client.list_recent_collections(0,10))"
  2. Check Document AI processing status in the h2oGPTe UI — look for ingestion errors
  3. Confirm SharePoint Graph API permissions are admin-consented in Azure Portal
📄 Document generation fails
  1. Check document generator logs:
    bash
    kubectl logs -l app=jbs-docgen -n certis-jbs --tail=50
  2. Verify S3 bucket exists and credentials are valid — test with aws s3 ls s3://certis-jbs-documents
  3. Confirm jbs_template.docx is present in the container image — check Dockerfile.document
💾 Session state lost after orchestrator pod restart

If STATE_BACKEND=memory, state is lost on restart by design. Switch to sqlite for restart-safe state:

  1. Set STATE_BACKEND=sqlite and SQLITE_PATH=/data/jbs_sessions.db in Helm values
  2. Mount a PersistentVolume at /data on the orchestrator pod so the SQLite file survives restarts
  3. For multi-replica deployments, use STATE_BACKEND=external_redis with a managed Redis URL (e.g. AWS ElastiCache) — do not deploy a Redis pod via Helm

Note: conversation turn history is never lost regardless of state backend — it lives in h2oGPTe natively via conversation_id.

🔄 SharePoint sync not running
  1. Check CronJob and recent job status:
    bash
    kubectl get cronjobs -n certis-jbs
kubectl get jobs -n certis-jbs
  2. Trigger a manual sync:
    bash
    kubectl create job --from=cronjob/certis-jbs-sharepoint-sync \
  manual-sync-$(date +%s) -n certis-jbs
