JBS Automation Platform

Deployment Guide

Deploy the Job Breakdown Statement platform on Azure Container Apps. Three middleware services receive Microsoft Teams messages, run a 2-phase AI interview via H2O Enterprise h2oGPTe, and deliver a signed Word document via Azure Blob Storage.

☁️ Platform: Azure Container Apps
💬 Messaging: Microsoft Teams / Azure Bot Service
🤖 AI Engine: H2O Enterprise h2oGPTe (HAIC)
📦 Storage: Azure Blob Storage
🔄 CI/CD: GitHub Actions → ACR → ACA
🗺
Overview
Platform Architecture

Five components make up the platform. The three middleware services run on ACA and communicate via internal DNS. h2oGPTe and the admin dashboard remain on HAIC.

Service / ImagePortIngressEntrypointHosting
jbs-webhook 8000 External src.webhook.main:app Azure Container Apps
jbs-orchestrator 8001 Internal src.agent.server:app Azure Container Apps
jbs-docgen 8002 Internal src.document.server:app Azure Container Apps
jbs-sp-sync — Job python -m src.rag.sharepoint_sync ACA Scheduled Job (02:00 UTC)
jbs-dashboard 10101 HAIC wave run dashboard/app.py HAIC App Store — not ACA

Message flow: Teams → Azure Bot Service → Webhook (/webhook/teams, validates RS256 JWT) → Orchestrator (/process, 2-phase state machine, calls h2oGPTe) → on APPROVE → Document Generator (/generate, renders .docx, uploads to Blob Storage) → Bot Framework REST API → reply in Teams.

2-phase interview: Phase 1 is code-only (5-step counter) — collects customer_name, site_name, site_category, job_purpose, authorized_by. Phase 2 is hybrid LLM+RAG — iterates duties+tasks sections then safety, presents a review summary, and on APPROVE triggers document generation. Approval keywords: approve, confirm, yes, proceed, looks good.

Internal service URLs (within the ACA Environment): http://jbs-orchestrator and http://jbs-docgen — no port required within the same environment.

Step 5
Register Azure Resource Providers

On a new Azure subscription, resource providers must be registered before use. This is a one-time operation per subscription.

bash
az provider register --namespace Microsoft.ContainerRegistry
az provider register --namespace Microsoft.Storage
az provider register --namespace Microsoft.KeyVault
az provider register --namespace Microsoft.App
az provider register --namespace Microsoft.OperationalInsights

Registration is asynchronous. Wait until all providers show Registered before proceeding:

bash
for ns in Microsoft.ContainerRegistry Microsoft.Storage Microsoft.KeyVault Microsoft.App Microsoft.OperationalInsights; do
  echo -n "$ns: "
  az provider show -n $ns --query registrationState -o tsv
done

Re-run the check until all five return Registered (typically 1–3 minutes each).

Step 6
Create and Deploy the Microsoft Teams Bot

The Webhook service validates RS256 JWT Bearer tokens issued by Azure Bot Service. The Orchestrator sends replies via the Bot Framework REST API using OAuth2 client credentials. This step covers creating the bot identity, enabling the Teams channel, building the app package, and publishing it to your tenant.

6a

Create the App Registration in Microsoft Entra ID

The bot needs an identity in Azure AD before the Azure Bot resource is created.

  1. Azure Portal → Microsoft Entra ID → App registrations → New registration
  2. Name: jbs-bot
  3. Supported account types: Accounts in this organizational directory only (single-tenant)
  4. Redirect URI: leave blank → Register
  5. On the overview page, copy:
    • Application (client) ID → this is TEAMS_APP_ID
    • Directory (tenant) ID → note for reference
  6. Go to Certificates & secrets → Client secrets → New client secret
    • Description: jbs-bot-secret
    • Expires: 24 months
    • Click Add → copy the Value immediately → this is TEAMS_APP_PASSWORD
⚠️

The client secret value is only shown once. Copy it before leaving the page.

  2. Fill in:

    • App ID: paste your TEAMS_APP_ID from Step 6a
  3. Review + create → Create
  4. Once deployed, navigate to the resource → Configuration and confirm the Microsoft App ID matches Step 6a

Publish the bot to Microsoft Teams

  3. The app appears in the list with status Blocked by default

Step 7
Build Docker Images

Build all images locally. No Azure credentials required. The orchestrator image bundles config/prompts/; the docgen image bundles templates/.

bash
# Webhook service — FastAPI on port 8000
docker build --platform linux/amd64 -f Dockerfile.webhook -t jbs-webhook:${IMAGE_TAG} .

# Orchestrator — FastAPI on port 8001 (bundles config/prompts/)
docker build --platform linux/amd64 -f Dockerfile.orchestrator -t jbs-orchestrator:${IMAGE_TAG} .

# Document generator — FastAPI on port 8002 (bundles templates/)
docker build --platform linux/amd64 -f Dockerfile.document -t jbs-docgen:${IMAGE_TAG} .

# H2O Wave admin dashboard — deployed via HAIC App Store, NOT pushed to ACR
docker build --platform linux/amd64 -f Dockerfile.dashboard -t jbs-dashboard:${IMAGE_TAG} .
⚠️

Apple Silicon note: The --platform linux/amd64 flag is required when building on M-series Macs. Azure Container Apps only accepts linux/amd64 images — omitting the flag produces an arm64 image that will be rejected at deploy time.

Confirm all images built:

bash
docker images | grep jbs

Test locally before pushing:

bash
docker compose up --build
# webhook: http://localhost:8000/health
# orchestrator: http://localhost:8001/health
# docgen: http://localhost:8002/health
💡

docker-compose.yml sets STATE_BACKEND=memory and wires ORCHESTRATOR_URL/DOCUMENT_GENERATOR_URL automatically. Requires a populated .env.

Step 9
Create Azure Blob Storage

The Document Generator uploads .docx files to Blob Storage and returns a 15-minute SAS URL to the user. The blob container name and key are read from environment variables AZURE_STORAGE_CONTAINER and AZURE_STORAGE_KEY.

bash
az storage account create --name $STORAGE_ACCOUNT --resource-group $RG --location $LOCATION --sku Standard_LRS --kind StorageV2 --allow-blob-public-access false
⚠️

DNS propagation: Wait ~30–60 seconds after the account is created before running the next commands. The blob endpoint (jbsstorage.blob.core.windows.net) may not resolve immediately.

Retrieve the storage key, then create the container using it:

bash
export STORAGE_KEY=$(az storage account keys list --account-name $STORAGE_ACCOUNT --resource-group $RG --query "[0].value" -o tsv)

# Verify the key was retrieved
echo $STORAGE_KEY

az storage container create --name jbs-documents --account-name $STORAGE_ACCOUNT --account-key $STORAGE_KEY

Update your .env:

bash
AZURE_STORAGE_ACCOUNT=jbsstorage
AZURE_STORAGE_KEY=<paste STORAGE_KEY value>

Step 10
Create Azure Key Vault & Store Secrets
bash
az keyvault create --name $KV_NAME --resource-group $RG --location $LOCATION --sku standard --enable-rbac-authorization true

The vault uses Azure RBAC for access control. Assign yourself the Key Vault Secrets Officer role before setting secrets:

bash
az role assignment create --role "Key Vault Secrets Officer" --assignee $(az ad signed-in-user show --query id -o tsv) --scope $(az keyvault show --name $KV_NAME --resource-group $RG --query id -o tsv)

Wait ~30 seconds for the role assignment to propagate, then set the four secrets:

bash
az keyvault secret set --vault-name $KV_NAME --name teams-app-password --value "<TEAMS_APP_PASSWORD>"
az keyvault secret set --vault-name $KV_NAME --name h2ogpte-api-key --value "<H2OGPTE_API_KEY>"
az keyvault secret set --vault-name $KV_NAME --name azure-client-secret --value "<AZURE_CLIENT_SECRET>"
az keyvault secret set --vault-name $KV_NAME --name storage-key --value "$STORAGE_KEY"

Verify all four secrets are stored:

bash
az keyvault secret list --vault-name $KV_NAME --query "[].name" -o tsv

Step 12
Deploy Services to Azure Container Apps

Deploy webhook (external ingress), orchestrator (internal), and document generator (internal). Container App names must be jbs-* to match the GitHub Actions workflow.

12a

Retrieve ACR credentials

bash
export ACR_USERNAME=$(az acr credential show --name $ACR_NAME --query username -o tsv)
export ACR_PASSWORD=$(az acr credential show --name $ACR_NAME --query "passwords[0].value" -o tsv)
12b

Deploy the Webhook service (external ingress — public)

bash
az containerapp create \
  --name jbs-webhook \
  --resource-group $RG \
  --environment $ACA_ENV \
  --image ${ACR_LOGIN_SERVER}/jbs-webhook:${IMAGE_TAG} \

  --registry-username $ACR_USERNAME \

  --target-port 8000 \
  --min-replicas 2 \
  --max-replicas 10 \
  --cpu 0.5 --memory 1.0Gi \

    ORCHESTRATOR_URL="http://jbs-orchestrator" \
  --secrets \
    teams-app-password="<TEAMS_APP_PASSWORD>"

Get the FQDN and set the Teams Bot messaging endpoint:

bash
export WEBHOOK_FQDN=$(az containerapp show \
  --name jbs-webhook \
  --resource-group $RG \
  --query "properties.configuration.ingress.fqdn" -o tsv)

echo "Teams Bot messaging endpoint: https://${WEBHOOK_FQDN}/webhook/teams"
⚠️

→ Paste this URL into Azure Portal → Azure Bot → Configuration → Messaging endpoint (Step 6c). The bot will not receive any messages until this is set.

12c

Deploy the Orchestrator service (internal ingress only)

bash

  --name jbs-orchestrator \
  --resource-group $RG \

    TEAMS_APP_ID="<TEAMS_APP_ID>" \
    DOCUMENT_GENERATOR_URL="http://jbs-docgen" \
    AZURE_TENANT_ID="<AZURE_TENANT_ID>" \
    AZURE_CLIENT_ID="<AZURE_CLIENT_ID>" \
    SP_SITE_URL="https://wigyoai.sharepoint.com/sites/h2O" \
    SP_LIBRARY_CORPORATE="<library-id>" \
    SP_LIBRARY_AVIATION="<library-id>" \
    SP_LIBRARY_INDUSTRIAL="<library-id>" \
    SP_LIBRARY_MARITIME="<library-id>" \
    SP_LIBRARY_RETAIL="<library-id>" \
    STATE_BACKEND="memory" \
    SESSION_TTL_HOURS="24" \

⚠️

Single worker required: The orchestrator must run with --workers 1 (set in its Dockerfile / entrypoint). Troubleshooting

Split-brain sessions — users get different responses on each message

Cause: The orchestrator is running with multiple uvicorn workers (--workers 2 or more). In-memory session state (STATE_BACKEND=memory) is process-local — each worker has its own isolated copy of the session store. Requests routed to different workers see different (or empty) session state.

Fix: Ensure the orchestrator Dockerfile / entrypoint uses exactly --workers 1:

dockerfile
CMD ["uvicorn", "src.agent.server:app", "--host", "0.0.0.0", "--port", "8001", "--workers", "1"]

Rebuild and redeploy the orchestrator after making this change.

"database is locked" error in orchestrator logs

Cause: STATE_BACKEND=sqlite was set and the SQLite file was placed on an Azure Files mount (SMB protocol). SMB does not support POSIX advisory locks; SQLite's WAL mode relies on them, causing lock contention errors immediately.

Fix: Set STATE_BACKEND=memory in the orchestrator env-vars. Remove any SQLITE_PATH variable and any Azure Files volume mounts from the container app definition:

bash
az containerapp update \
  --name jbs-orchestrator \
  --resource-group $RG \
  --set-env-vars STATE_BACKEND=memory \
  --remove-env-vars SQLITE_PATH

Stale Teams re-deliveries resetting live sessions

Cause: Microsoft Teams occasionally re-delivers old Bot Framework Activity events (retry on perceived timeout). If the orchestrator processes these stale messages it re-runs earlier state-machine steps, resetting an in-progress session.

Fix: The orchestrator applies a timestamp age filter — any incoming message whose timestamp is more than 10 seconds old on an already-active session is silently dropped. Verify this filter is present in src/agent/server.py. If sessions are still being reset, check whether the 10-second threshold needs to be increased for high-latency environments.

Word template placeholders not replaced in generated .docx

Cause: The original _replace_placeholders() helper only iterated top-level paragraphs in the document body. Placeholders inside table cells (used for the 6-field metadata table in the JBS template) were never reached and remained as literal {{field}} strings in the output.

Fix: The _all_paragraphs() helper now traverses both body paragraphs and all paragraphs inside table cells. _replace_in_paragraph() operates on individual runs to preserve character formatting. Confirm the fix is present in src/document/generator.py before deploying.

GitHub Actions CI/CD fails at azure/login step

Cause: azure/login@v2 dropped support for passing credentials as individual parameters (client-id, tenant-id, client-secret as separate inputs). Using that form produces an authentication error silently or with a misleading message.

Fix: Pass all credentials as a single JSON object via the creds input. Add AZURE_SUBSCRIPTION_ID as a GitHub Actions repository secret, then update the workflow step:

yaml
- uses: azure/login@v2
  with:
    creds: |
      {
        "clientId": "${{ secrets.AZURE_CLIENT_ID }}",
        "clientSecret": "${{ secrets.AZURE_CLIENT_SECRET }}",
        "subscriptionId": "${{ secrets.AZURE_SUBSCRIPTION_ID }}",
        "tenantId": "${{ secrets.AZURE_TENANT_ID }}"
      }

Required GitHub secrets: AZURE_CLIENT_ID, AZURE_CLIENT_SECRET, AZURE_SUBSCRIPTION_ID, AZURE_TENANT_ID, ACR_LOGIN_SERVER, AZURE_RESOURCE_GROUP.
